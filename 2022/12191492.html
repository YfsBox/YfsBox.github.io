1. Tiger编译器的栈帧

Temp

Temp是在Tiger中关于寄存器的抽象.其中的数据成员只有一个,即为num.表示的是寄存器的编号.

struct myTemp_  { int num; };

此外还有一个用于分配寄存器时的全局变量,定义如下:

static int g_tempCount = 0;

当调用newTemp时:

myTemp 	Temp_newTemp(void)
{
    myTemp temp = makeMemoryBlock(sizeof(*temp), MEMORY_TYPE_NONE);
    assert (temp);
    temp->num = ++g_tempCount;
    return temp;
}

每次都对该全局变量+1,并且设置该Temp中的num值.

Frame和Access

这一部分最核心的两种数据结构是Frame和Access.前者是对一个过程(函数)的抽象,其中包含了一个函数中所需要的参数,局部变量及其label等信息.其中核心的定义与方法如下:

typedef struct myFrame_	    	*myFrame;
struct myFrame_ // 是一种对于过程的抽象
{
    myAccessList    formals;            // 当前过程中的参数
    myString        viewShift;          
    int             localCount;         //  not include `formalLocalCount`      // 局部变量的计数值
    int             formalLocalCount;   //  in frame formal local count         // 形参数的计数值
    myLabel         label;          // 这个过程(函数)的label
};

其中viewShift用于处理后面提到的视角移位问题,当需要new出一个新的Frame对象时,所调用的方法为:

myFrame Frame_newFrame(myLabel frameLabel, myBoolList formalFlags)
{
    myFrame frame = makeMemoryBlock(sizeof(*frame), MEMORY_TYPE_NONE); // 底层封装了melloc
    assert (frame);
    frame->label = frameLabel; // 设置lable
    fillFormalsFromFlags(frame, formalFlags);
    fillViewShiftFromFlags(frame);
    frame->localCount = 0;  //  no include formal locals	// 目前还没有包含局部变量
    return frame;
}

我们可以发现,再new出一个Frame的时候没有对内部关于local的部分,进行设置,或者说,默认其中localCount为0,没有一个局部变量.那么局部变量是什么时候被加入到Frame中的呢?

此外,除了Frame,还有一个重要的数据结构为Access,该类型用来描述可以放入到栈中或者寄存器中的形参或者局部变量.其中详细的定义如下:

struct myAccess_  {
    enum AccessKind { InFrame, InReg } kind;
    union
    {
        int     offset; //  offset comapred to frame pointer
        myTemp  reg;    //  register representation
    }u;
};

由此我们可以发现,要么处于再寄存器中,其中myTemp是这个项目中对于寄存器的抽象,要么处于内存中(栈帧)中,其中offset表示的是在当前栈帧中的偏移量.其中关于Access的构造函数定义如下:

static myAccess makeInFrameAccess(int offset)
{
    myAccess access = makeMemoryBlock(sizeof(*access), MEMORY_TYPE_NONE);
    assert (access);
    access->kind = InFrame;
    access->u.offset=  offset;
    return access;
}

static myAccess makeInRegAccess(myTemp reg)
{
    myAccess access = makeMemoryBlock(sizeof(*access), MEMORY_TYPE_NONE);
    assert (access);
    access->kind = InReg;
    access->u.reg = reg;
    return access;
}

基本上就是比较普通set操作的函数,但是我们可以发现这两个函数都是static类型的,因此不是对外开放的接口(private).但是该函数被用于public的接口Frame_allocateLocal和fillFormalsFromFlags.这两个函数涉及到了形参或者局部变量的设置.后者在Frame的构造函数中被调用,因此我们可以说当一个Frame被构造的时候,就需要处理好关于Formal的问题,其中主要问题在于:(1)应当存储在寄存器里还是栈帧存储单元中.(2)如何处理视角移位问题.

至于局部变量的处理,则需要继续在递归AST时,进入到函数内部,每当扫描到一个局部变量就需要调用allocLocal来分配.其中该函数定义如下:

myAccess Frame_allocateLocal(myFrame frame, bool escapeFlag)
{
    if (escapeFlag)
        return makeInFrameAccess(-(++frame->localCount) * BASE_SIZE);
    else    
        return makeInRegAccess(Temp_newTemp());
}

其中需要区分的是该局部变量是否是逃逸的.每当扫描到一个变量的声明时,都需要调用该函数分配一个Access单元,除此之外还要该Access与变量的名字绑定.离开当前作用域时,该关联将会取消(因为离开作用域时,该变量的name就失效了,结合前面语义分析的实现),但是被分配过的空间仍然保留.因为离开一个作用域并不等于当前Frame实效.

Escape

对于局部变量和形参是否应当是逃逸的,我们需要专门的处理过程去处理.在Tiger中所采取的方法是递归地扫描AST.在扫描的过程中对于扫描到的变量判断其是否时逃逸的.由于Tiger将语义分析和代码生成放在了一起,因此需要在该阶段之前就完成excape的处理.

其中关于递归检查的模式仍然类似于语义分析的部分,仍然是递归地实现,但是相对来说所要做的处理没有那么复杂.这个过程中也涉及到环境表的操作.其中相关的实现如下:

void Escape_findEscape(myExp exp);
bool Escape_isVarEscaped(mySymbol varName);
myTable Escape_getEscapeEnvironment(void);
void Escape_setEscapeEnvironment(myTable escapeEnv);

Escape_findEscape用于从该exp开始向下递归地处理escape问题,而Escape_isVarEscaped用来从表中查询该name对应的变量是否时逃逸的.Escape_getEscapeEnvironment返回检查逃逸所用的环境表,Escape_setEscapeEnvironment用来设置环境表.

其中核心的函数如下,就类似于语义分析的实现,对于Dec,Var,Expr等类型的结点都有一个用于判断escape的,递归的实现.如下:

void Escape_findEscape_Exp(int depth, myExp exp);
void Escape_findEscape_Dec(int depth, myDec dec)
{
    switch (dec->kind)
    {
        case VarDec:
            return Escape_findEscape_VarDec(depth, dec->u.varDec);
        case TypeDec:
            return Escape_findEscape_TypeDec(depth, dec->u.tyDec);
        case FuncDec:
            MySymbol_BeginScope(Escape_getEscapeEnvironment());
            Escape_findEscape_FuncDec(depth + 1, dec->u.funcDec);
            MySymbol_EndScope(Escape_getEscapeEnvironment());
            return;
        default:        assert (false);
    }
}

其中有两类实现,第一类类似于一种分派器,其中接受的是像myExp,myDec抽象的类型而不是具体的类型,在内部的逻辑中会检查其类型并且调用其特定的处理函数.比如上面的关于Dec的处理,分为了VarDec,TypeDec,FuncDec三种类型其中关于FuncDec的处理有一点特殊,涉及到了环境表的操作,并且调用具体的判断时还需要将depth+1.

第二类也就是对某种具体类型的escape处理函数.其实现如下:

void Escape_findEscape_FuncDec(int depth, myFuncDec funcDec)
{
    myTyFieldList formalFields;
    myExp body;
    switch (funcDec->kind)
    {
        case FunctionDec:
        {
            //  assume all formals escapes
            formalFields = funcDec->u.functionDec->tyFieldList;
            body = funcDec->u.functionDec->exp;
            break;
        }
        case ProcedureDec:
        {
            //  assume all formals escapes
            formalFields = funcDec->u.procedureDec->tyFieldList;
            body = funcDec->u.procedureDec->exp;
            break;
        }
        default:        assert (false);
    }
    treatFormalsAsNotEscapedVars(depth, formalFields);
    Escape_findEscape_Exp(depth, body);
}

每当在静态嵌套深度为depth的函数中发现了变量或者形参的声明时,就创建一个环境表所对应的项(其中escape域设置为false),并且将其加入到环境表中.之后在新的环境中处理该作用域中的表达式时,如果发现了使用了之前声明过的变量,由于当前的depth比当时声明时创建的更大,因此会将其设置为true.

static bool isVarNestedUsed(int depth, mySymbol varSymbol)
{
    EscapeEntry entry = Escape_getVarEntry(varSymbol);
    assert (depth >= entry->depth);
    if (entry->depth == depth)
        return false;
    else
        return true;
}

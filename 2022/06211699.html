Word Count:   5.1k
Read Times:   20 Min

1.基本概念

1)物理内存布局

在之前,学习完boot部分就可以知道,内核代码被加载到了从物理内存0开始的位置.接下来的物理内存分布状况,大概是:高速缓存区,虚拟盘,主内存区.

其中高速缓冲区用于,当进程需要读取块设备是,系统首先会将数据读到高速缓存(忽略其中640K--1M的BIOS ROM),而需要写入块设备时,先放到高速缓存区,然后再由块驱动设备写到设备中.

所以说,高速缓存区主要充当主存和块设备的缓存.

主内存区是提供给进程申请或者使用的内存区,在内存申请或者释放的过程中,需要内核区代码的介入.

2)内存地址空间

主要概念有:虚拟地址与逻辑地址,线性地址和物理地址.

虚拟地址由段选择符和段内偏移组成,需要经过地址翻译才能得到物理地址.甚至可以说,虚拟地址一部分藏在段寄存器里也就是(段选择符),另一部分一般是在程序中生成的逻辑地址.

虚拟地址空间的定义是由GDT和LDT决定的.GDT和LDT在linux 0.11中是虚拟地址到线性地址的桥梁.主要体现于,借此将段寄存器中的值(段选择符)转化成了一个段基地址.

而逻辑地址则是虚拟地址中的偏移部分,也是应用程序直接接触的地址(很多地方不区分虚拟地址和逻辑地址).

线性地址指的则是段基地址和段内偏移.在不使用分页机制中,线性地址就是物理地址.

虚拟存储空间为进程提供了一种巨大,稀疏的假象,其核心实现思想就是将主存当作磁盘的缓存,根据按需分配的思想.更具体地说,当进程需要使用的内存不在主存中,CPU通过缺页异常得知,并将线性地址放入cr2寄存器,进而缺页处理程序可以将需要的页面从磁盘加载到主存.

实模式和保护模式最大的区别在于:

  • 实模式通过直接段寄存器+偏移值即可得到地址.
  • 而保护模式中的段寄存器表示的是段描述符索引.段描述符中也会增加一些和保护相关的信息.

其中描述符表有,描述符主要用于寻址,并记录一些与保护相关的信息:

  • GDT,可以被任何程序用于引用访问一个内存段.
  • IDT,处理中断时,用于寻址到中断处理程序.
  • LDT,每个任务都有一个LDT,可以视为GDT的扩充,为每个任务提供可寻址内存空间的范围.

描述符表是完成虚拟地址到线性地址映射的关键,它们的地址都被加载到特定的寄存器中

我们可以发现在linux 0.11中,GDT和LDT呈现一种包含的关系,GDT中除了代码段数据段等,主要由一个个LDT和状态段组成.

LDT在每个task的数据结构中都有,而TSS用于保存每个task的上下文,在面临进程切换时使用.

除此之外,还有个地方Linux 0.11中,内核和任务的代码段和数据段是重叠的.

将分页模式开启,需要将CR0的最高位设置为1.页表是位于内核空间的一种数据结构,页表的作用在于记录线性地址和物理地址的映射.在linux 0.11中,内核与所有任务都是使用同一个目录表的,所以线性地址到物理地址的映射函数都一样,为了防止冲突,需要将虚拟地址到线性地址的映射不同.

3)多任务与保护机制

linux 0.11中只用了0和3两个保护级

内核代码本身有所有任务共享,保存在GDT中.

每个任务都有属于自己的代码和数据区,这两个区域保存在LDT中,这为不同的任务提供了隔离性.

每个进程都有自己的内核栈,当运行此进程需要进入内核态时,会运行在此进程的内核栈中.

5)写时复制与需时加载

写时复制

是一种推迟或者免除数据复制的方法.当进程A通过fork创建出来一个新的子进程B的时候,默认对于内存空间是共享的,对于B也是以只读的方式访问内存空间,父进程也是只读.当两者中有需要写的时候,就会触发缺页异常,会执行do_wp_page()来解决问题.

更具体地说,do_wp_page()对需要写入的物理页,取消共享(up_wp_page()).为写进程复制一个新的物理页,两者对于这块物理页都有相互独立的一份.最后,修改权限位.

返回中断处理时,重新执行写操作指令.

所以总体来说,能够减少一些不必要的数据复制的开销.

2.源码分析

1)主存区初始化

首先,我们在初始化的main函数中,首先看到有一段代码,用来计算物理内存中比较关键的几个基地址的操作.

	memory_end = (1<<20) + (EXT_MEM_K<<10);
	memory_end &= 0xfffff000;
	if (memory_end > 16*1024*1024)
		memory_end = 16*1024*1024;
	if (memory_end > 12*1024*1024) 
		buffer_memory_end = 4*1024*1024;
	else if (memory_end > 6*1024*1024)
		buffer_memory_end = 2*1024*1024;
	else
		buffer_memory_end = 1*1024*1024;
	main_memory_start = buffer_memory_end;
#ifdef RAMDISK
	main_memory_start += rd_init(main_memory_start, RAMDISK*1024);
#endif

其中memory_end用于表示内存的边界地址,最大为16MB,除此之外再根据最大地址的状况,设置缓冲区地址,最后设置主内存起始位置.总之这三者 mempry_end,buffer_memory_end,main_memory_start的地址分布状况,和前面描述的物理内存分布基本一样.

随后就进入 mem_init进行初始化,这个函数主要用来初始化主内存区的,其中参数分别为主存的起始和末尾地址.

void mem_init(long start_mem, long end_mem)
{
	int i;
	HIGH_MEMORY = end_mem;
	for (i=0 ; i<PAGING_PAGES ; i++)
		mem_map[i] = USED;
	i = MAP_NR(start_mem);
	end_mem -= start_mem;
	end_mem >>= 12;
	while (end_mem-->0)
		mem_map[i++]=0;
}

首先我们来看一下mem_map这个数据结构.

static unsigned char mem_map [ PAGING_PAGES ] = {0,};

这个数据结构的作用相当于ucore中的free_list.其中每一项所对应的物理内存的关系是固定的.

这个数据结构只是内核用来记录物理内存的使用状态的,是空闲空间管理的基础,其中的作用范围在于1MB以上的内存(内核代码和数据之外).每一项的数值指的是此物理被占用的次数.这个数据结构起到linux 0.11中的空闲空间管理的作用,相比ucore使用 free_list,这个数据结构如果想要展开使用一些页分配的调度算法,比如说 first fit,best fit等,相比list是比较不方便的.所以这个地方体现了linux 0.11在内存管理上是比ucore要简陋的.

所以总体上来看,mem_init所做的工作,主要是将主内存区对应的 mem_map设置为0(未使用的).执行完此函数后,最终的 mem_map效果如下:

到这里,如果想要比较深入地理解内存管理还是不够的,最好再看一下关于页表(页目录表)以及一些相关内核API的实现方式.

关于页表和页目录表的定义,回忆之前的boot部分,需要记住的是,它们在内存中的范围是0-5000的.但是相比于ucore来说,linux 0.11有关于页表的实现是比较简陋的,ucore中会有相关的数据结构将页表和页表项进行抽象.

2)对于有关于页与页表的内核API

对于页和页表的一些内核API,也应该注意各API之间也有不同的层次感和抽象.下面的这几个相对而言get_free_page,free_page,cpoy_page,是比较底层的实现,并且抽象层次较低(基于物理地址的)的.后来的一些抽象层次更高的API,很多都是基于下面的这几个API实现的.


#define copy_page(from,to) \
__asm__("cld ; rep ; movsl"::"S" (from),"D" (to),"c" (1024))
unsigned long get_free_page(void)
{
register unsigned long __res asm("ax");
__asm__("std ; repne ; scasb\n\t"
	"jne 1f\n\t"
	"movb $1,1(%%edi)\n\t"
	"sall $12,%%ecx\n\t"
	"addl %2,%%ecx\n\t"
	"movl %%ecx,%%edx\n\t"
	"movl $1024,%%ecx\n\t"
	"leal 4092(%%edx),%%edi\n\t"
	"rep ; stosl\n\t"
	"movl %%edx,%%eax\n"
	"1:"
	:"=a" (__res)
	:"0" (0),"i" (LOW_MEM),"c" (PAGING_PAGES),
	"D" (mem_map+PAGING_PAGES-1)
	);
return __res;
}
void free_page(unsigned long addr)
{
	if (addr < LOW_MEM) return;
	if (addr >= HIGH_MEMORY)
		panic("trying to free nonexistent page");
	addr -= LOW_MEM;
	addr >>= 12;
	if (mem_map[addr]--) return;
	mem_map[addr]=0;
	panic("trying to free free page");
}

copy_page的功能在于给定 from和 to的物理地址,实现内存的拷贝,底层直接调用汇编,使用了“rep循环+mov”的模式,这种方式在boot的内存移动中也有所使用.到这里,对于copy_page这个API,是建立在from和to所对应的物理页面都已经分配好并且有效的前提上的.

get_free_page总体来说是一种 first fit页分配算法,从 mem_map的末尾开始扫描,直到扫到一个空闲页就返回页地址(物理页地址).因此到这里也可以推知,后来需要分配内存块时,其在微观上是以一个物理页为单位,逐页逐页地找出内存空间的.这也就体现了,在虚拟内存空间连续的内存块,在物理空间上未必是连续的

free_page,释放物理地址 addr中的一页内存,除了判断是否出现地址异常之外,其核心部分无外乎对 mem_map中对应物理页的引用计数的修改.其实这对于完整意义上的内存页释放来说,根本就是不够的,还需要后面的关于page_table的操作,这样在free_page_tables这个API中有所体现.

总体来说,上面的这三个API是关于内核处理内存中,最最基础而底层的,只是涉及到物理地址,并且只是以一物理页为单位进行操作.

然而物理内存空间和线性内存空间的映射是如何建立的呢?

在此之前,需要明确的是page_table是线性地址到物理地址的桥梁.

答案就藏在 put_page中.这个用于将一个给定的线性地址绑定到一个物理页上.落实在对页表(页目录表)的设置上.

unsigned long put_page(unsigned long page,unsigned long address)
{//其中的参数指的是某一个物理页的地址,address则是要求进行绑定的线性地址
	unsigned long tmp, *page_table;
	if (page < LOW_MEM || page >= HIGH_MEMORY)//物理页地址不在主内存区的范围中
		printk("Trying to put page %p at %p\n",page,address);
	if (mem_map[(page-LOW_MEM)>>12] != 1)//如果已经申请
		printk("mem_map disagrees with %p at %p\n",page,address);
	page_table = (unsigned long *) ((address>>20) & 0xffc);//线性地址的前十位得到页目录表项的基地址
	if ((*page_table)&1)//如果页目录表存在
		page_table = (unsigned long *) (0xfffff000 & *page_table);//取出对应页表项地址
	else {
		if (!(tmp=get_free_page()))//如果不存在就先分配一个物理页,tmp得到该物理页的物理地址
			return 0;//这个物理页是为了这个目录表所分配的
		*page_table = tmp|7;
		page_table = (unsigned long *) tmp;//将此页目录表与新分配的物理页绑定
	}
	page_table[(address>>12) & 0x3ff] = page | 7;//然后再将此页目录下的页表项设置为这个page.此page与address的绑定正式完成.
/* no need for invalidate */
	return page;
}

简而言之,过程如下:

  • 检查条件,是否越界或者是否此页已经被占用.
  • 得出目录项地址.
  • 对于目录项有效(有物理页与该目录项绑定):直接得出页表项的基地址.
  • 如果无效,先分配一个物理页得出其物理地址tmp,根据此物理地址填充页目录项内容,tmp也就是该目录下所有页表项的基地址.然后
  • 根据address中页表索引定位到页表项,并根据page填写其内容.

这个过程有点绕,但总的来说:

  • 如果页目录表本身就持有有效的页表项(p == 1):定位页目录项地址--->根据页目录项内存定位目录下页表的基地址--->根据address定位二级页表地址,并填写其内容.
  • 如果 P == 0,那么“根据address定位页目录项地址--->分配二级页表项所占的物理页--->将二级页表物理页与该目录项绑定--->从新分配的tmp中由address检索出页表项,并填写内容”

总体来说,这个地方和ucore的实现基本上是一致的.

然后再来分析一下关于清除页面内容,同时需要释放相关的页表项的函数.

int free_page_tables(unsigned long from,unsigned long size)//这里的from依旧是线性地址,size是字节数
{
	unsigned long *pg_table;
	unsigned long * dir, nr;
	if (from & 0x3fffff)
		panic("free_page_tables called with wrong alignment");
	if (!from)
		panic("Trying to free up swapper memory space");
	size = (size + 0x3fffff) >> 22;//此时求出来的是,这片内存所跨越的目录数
	dir = (unsigned long *) ((from>>20) & 0xffc);//得出第一个目录地址
	for ( ; size-->0 ; dir++) {//外层循环表示逐个目录
		if (!(1 & *dir))//无效目录表示,这个目录内没有分配的物理页,直接越过
			continue;
		pg_table = (unsigned long *) (0xfffff000 & *dir);//得出来这个目录所对应的二级页表项的页的基地址
		for (nr=0 ; nr<1024 ; nr++) {//对于这个目录下的页,逐页释放.
			if (1 & *pg_table)
				free_page(0xfffff000 & *pg_table);
			*pg_table = 0;//页表设置为0.
			pg_table++;//下一页
		}
		free_page(0xfffff000 & *dir);//将该目录所在的页清空
		*dir = 0;
	}
	invalidate();
	return 0;
}

如果是直接清空对应的物理内存页的话,直接从from开始逐个释放掉就可以了.如果仍然是考虑从from逐个地遍历并且顺便将其页表,页目录表释放掉将会带来问题,因为线性地址到物理地址的映射本身就是依赖于页表和页目录表的,所以我们在进行内存释放时,不应该忽略这种依赖关系,否则将会在后续的遍历中打乱.

而在linux 0.11中,使用了一种自顶向下的方式,首先确定内存释放时,所跨越的目录的范围,然后逐个目录地释放,每个目录中,逐页地释放.每次释放都是落实在free_page+*pte = 0上.

简而言之:

for each dir{
  	if 目录无效: continue;
  	for each page{
      释放物理页,页表项设置为0.
    }
    释放目录物理页,目录项设置为0.
}

下面是有关于拷贝物理页的分析.

int copy_page_tables(unsigned long from,unsigned long to,long size)//此时from和to都是线性地址
{
	unsigned long * from_page_table;
	unsigned long * to_page_table;
	unsigned long this_page;
	unsigned long * from_dir, * to_dir;
	unsigned long nr;

	if ((from&0x3fffff) || (to&0x3fffff))
		panic("copy_page_tables called with wrong alignment");
	from_dir = (unsigned long *) ((from>>20) & 0xffc); 
	to_dir = (unsigned long *) ((to>>20) & 0xffc);
	size = ((unsigned) (size+0x3fffff)) >> 22;//分别得到目录表地址和size(所占的目录数量)
	for( ; size-->0 ; from_dir++,to_dir++) {
		if (1 & *to_dir)
			panic("copy_page_tables: already exist");
		if (!(1 & *from_dir))
			continue;
		from_page_table = (unsigned long *) (0xfffff000 & *from_dir);//将目录表中的内容提取前20位,前20位对应的是页祯地址
    if (!(to_page_table = (unsigned long *) get_free_page()))//为to的目录分配物理页
			return -1;	/* Out of memory, see freeing */
		*to_dir = ((unsigned long) to_page_table) | 7;//根据分配的物理页,设置to的目录项
		nr = (from==0)?0xA0:1024;//如果from为0,则说明来自内核,否则就是正常的主存区内存.
		for ( ; nr-- > 0 ; from_page_table++,to_page_table++) {//此时from_page_table是该目录下的第一个页的地址
			this_page = *from_page_table;//得到的是from的某个页表项
			if (!(1 & this_page))
				continue;
			this_page &= ~2;//忽略后面的权限位
			*to_page_table = this_page;//将from的页表项的内容拷贝到了to对应的页表项中.
			if (this_page > LOW_MEM) {//??
				*from_page_table = this_page;
				this_page -= LOW_MEM;
				this_page >>= 12;
				mem_map[this_page]++;
			}
		}
	}
	invalidate();
	return 0;
}

对于页的拷贝,仍然是既涉及到页表和物理页内容的拷贝.由于物理页的定位依赖于页表内容,所以进行拷贝时不能打破这种依赖关系,所以在此在页的拷贝上仍然采取了和free_page_tables一样的自顶向下的递归方法.

总体来说,过程可以概括如下:

  • 前置处理,得出from和to的目录地址,和size(所占的目录数量).
  • 进入一个二重循环,外围循环代表每一个目录的迭代,内层循环代表的是目录内每个页的迭代.
  • 初始化tlb.

3)缺页处理程序

这一部分的定义在page.s中,使用纯汇编实现.

中断向量表的对应项的设置,早在trap.c中就给予设置了.

set_trap_gate(14,&page_fault);

这一部分栈中寄存器的映像和一般的其他中断还是略有区别的.

	popl %eax
	iret

我们首先来回顾一下,相比于一般的中断调用含有出错号的情况,并没有压入一个函数地址,压入的通用寄存器数量也是较少的.最最核心的部分在于将出错的线性地址和错误类型压入,作为进一步调用函数的参数.要么是执行do_no_page要么是do_wp_page.这对应了两种缺页的情况:

  • 确实缺页,对于该线性地址,主存中没有一个物理页与之绑定.
  • 写页错误,这个与后面的写时复制有关.

下面来具体的分析一下这两个函数.

首先看do_no_page的处理:

void do_no_page(unsigned long error_code,unsigned long address)
{
	int nr[4];
	unsigned long tmp;
	unsigned long page;
	int block,i;
	address &= 0xfffff000;
	tmp = address - current->start_code;//我们需要明确的一点是,当发生中断时,current移动就是触发该中断的进程.
	if (!current->executable || tmp >= current->end_data) {
		get_empty_page(address);
		return;
	}
	if (share_page(tmp))
		return;
	if (!(page = get_free_page()))
		oom();
//下面的操作是与磁盘的读取有关的.
	block = 1 + tmp/BLOCK_SIZE;
	for (i=0 ; i<4 ; block++,i++)
		nr[i] = bmap(current->executable,block);
	bread_page(page,current->executable->i_dev,nr);
	i = tmp + 4096 - current->end_data;
	tmp = page + 4096;
	while (i-- > 0) {
		tmp--;
		*(char *)tmp = 0;
	}
	if (put_page(page,address))
		return;
	free_page(page);
	oom();
}

其中的参数分别是错误码和线性地址,分别对应压栈过程中的%eax和%edx.

void un_wp_page(unsigned long * table_entry)
{
	unsigned long old_page,new_page;

	old_page = 0xfffff000 & *table_entry;
	if (old_page >= LOW_MEM && mem_map[MAP_NR(old_page)]==1) {
		*table_entry |= 2;
		invalidate();
		return;
	}
	if (!(new_page=get_free_page()))
		oom();
	if (old_page >= LOW_MEM)
		mem_map[MAP_NR(old_page)]--;
	*table_entry = new_page | 7;
	invalidate();
	copy_page(old_page,new_page);
}
